# PIIvacy — How This Was Built

> Everything that happened, in one document. The build story of `piivacy`,
> a serverless PII scrubber for LLM calls, plus its public-facing demo at
> [piivacy.dev](https://piivacy.dev).

**TL;DR**: Started as "mostly-no-pii", a regex-driven scrubber spec. Grew
into PIIvacy — a fully tested npm package, a 267k-name fake-name database,
a tabbed marketing/demo site with in-browser LLM features, GitHub Actions
auto-deploy, AI-search optimization, a Resend-backed contact form, and a
custom domain. Built end-to-end in a single multi-day Cursor agent session.

## What it does (one paragraph)

PIIvacy is two functions. `scrub()` swaps PII out of text for safe
substitutes before the text goes to an LLM. `restore()` puts the real
values back when the LLM responds. Three substitution modes
(`token`, `realistic`, `pass-through`), per-call/per-category/per-label
configurable. 37 secret patterns, 21 other categories, with format
validators (Luhn, IBAN mod-97, SSN range, VIN check-digit, IPv4 octets,
real calendar dates). Optional opt-in BYO-LLM helpers for missed-PII
detection and intent-driven mode picking — the package never makes
HTTP calls itself; the user wires up whatever chat client they already
have. Hard-coded safety override: secrets/financial/identifiers
categories cannot be `pass-through`, even on explicit request.

## The chronological build

### Phase 1: Spec and core package (the "mostly-no-pii" era)

- Started from a one-paragraph design brief: dead-simple SDK, two functions,
  serverless, plain ESM, zero deps. Three substitution modes (token,
  realistic, pass-through). Pluggable patterns. BYO-LLM second-pass.
- Decided to build at workspace root as a sibling to abstraKt, not inside
  `src/`, so it could be trivially extracted to its own GitHub repo.
- Initial directory: `mostly-no-pii/` (the original name). The "mostly"
  framed honest expectations — regex misses things; defense in depth.
- Built core modules in order: `patterns.js` → `registry.js` → `modes.js`
  → `fake-values.js` → `sessions.js` → `scrub.js` → `restore.js` →
  `llm-check.js` → `llm-intent.js` → `adapters/`. Plus `index.js`.
- Wrote 18 test files with 191 initial assertions covering every public
  surface, error path, and known edge case.
- Validators: Luhn for credit cards, mod-97 for IBANs, SSA-style range
  for SSNs, ISO 3779 check-digit for VINs, octet bounds for IPv4, real
  calendar checks for DOB.

### Phase 2: The 267k-name table

- Realistic-mode NAME redaction needs a static lookup table. Sourced from
  US public-domain data:
  - **104,819 first names** from US SSA Baby Names (1880-present)
  - **162,253 surnames** from US Census 2010 Surnames file
- Wrote `scripts/build-names.mjs` to download, parse, bucket, and emit
  `data/names.json` (~23 MB).
- Bucketing strategy:
  - First names → `(sex, decade-of-peak-popularity)`. Filtered each name
    to its dominant sex to prevent rare-male "Sarah" entries leaking into
    male buckets.
  - Surnames → dominant Census ethnic distribution (white, eastAsian,
    southAsian, hispanic, black, native, mixed). Threshold >50% of each
    name's reported demographics.
- **First attempt failed**: tried OpenAI `text-embedding-3-small` for
  nearest-neighbor within bucket. Result was unusable — text-embedding
  models cluster name tokens by orthographic stem, so Marcus mapped to
  "Marquis", "Marquise", "Marrio". Spelling variants of itself, exactly
  the failure mode we were trying to avoid.
- **Second attempt worked**: frequency-rank with hash-derived offset
  inside each bucket, plus a Levenshtein-distance ≥4 filter. Marcus →
  Terrance, Bryant, Orlando, Cedric, Terrell. Garcia → Nunez, Valdez,
  Santiago, Maldonado, Dominguez. The right answer was demographic
  bucketing + frequency, not vector search.

### Phase 3: The interactive demo (mostly-no-pii-demo)

- Built `mostly-no-pii-demo/` as a sibling project with Express + plain
  HTML/CSS/JS. No framework.
- 500-sample test corpus: a build-time generator that creates 500 PII-laden
  realistic strings (customer support tickets, code/configs, chat,
  forms, logs, healthcare records, e-commerce orders, local searches,
  meeting notes, email threads). All values are FAKE (Visa test cards,
  555-01XX phones, RFC-6761 reserved domains) but match the regex
  patterns realistically.
- API surface: `/api/scrub`, `/api/scrub-all`, `/api/restore`,
  `/api/sample`, `/api/patterns`, `/api/llm/*` (4 endpoints for the
  WebLLM round-trip), `/healthz`, `/api/contact`.
- Dockerfile (Node 20 Alpine, non-root user, generates samples at
  build time, healthcheck at /healthz).
- Vendored the package via `cp -R ../mostly-no-pii vendor/mostly-no-pii`
  initially. Later replaced with a git dependency for cleaner sync.
- Deployed to Railway via `railway init` + `railway up`.

### Phase 4: Comprehensive UI rebuild

- The first cut of the demo got real feedback: "Stop pasting PII into LLMs"
  was the wrong framing (the whole point is you CAN paste it in safely),
  long-scroll layout was light on info, the round-trip restore demo was
  buggy, the visual aesthetic looked AI-generated.
- Rewrote with:
  - **Headline**: "Go ahead. Paste it in." with cyan-glowing emphasis.
  - **Tabbed SPA-lite**: 7 tabs (Demo, Modes, Patterns, LLM, Names, Docs,
    About) under a sticky backdrop-blur top nav. URL-hash driven.
  - **Linear/Vercel aesthetic**: Inter + JetBrains Mono via Google Fonts.
    8px spacing rhythm. Subtle gradients. Real elevation hierarchy. Custom
    dark scrollbars matching the theme.
  - **Generated illustrations**: 4 diagrams via OpenAI gpt-image-2 (routed
    through OpenRouter as `openai/gpt-5.4-image-2` because direct OpenAI
    access required org verification we didn't have). Hero scrubber-pipeline,
    three-mode comparison, LLM round-trip flow, names-bucketing diagram.
  - **OG/Twitter preview image**: 1200×630 generated via gpt-image-2,
    sips-cropped from 1536×1024 source.

### Phase 5: WebLLM in-browser LLM demos

- Two interactive demos that showcase the BYO-LLM helpers running entirely
  in the visitor's browser via WebLLM (`@mlc-ai/web-llm` from CDN):
  1. **Missed-PII detection** — text → regex scrub (pass 1) → ask the local
     LLM what's missed → re-scrub (pass 2)
  2. **Intent-driven mode picking** — query → local LLM picks per-category
     redact/preserve/synthetic decisions → scrub with those opts
- Initial model choice: Phi-3.5-mini-instruct (~2.4 GB). Too big for a
  voluntary download. Switched to Llama-3.2-1B (~700 MB) — smaller but
  anchored too hard on prompt examples and produced inconsistent JSON.
  Final choice: **Qwen2.5-1.5B-Instruct** (~950 MB) — purpose-tuned for
  structured-output instruction following, IndexedDB-cached after first
  download.
- Prompts include three contrasting few-shot examples each, so the model
  doesn't anchor on a single all-redact template.
- Hard-coded safety override in `parseScrubIntentResponse`: even if the
  LLM says `secrets: preserve` it gets downgraded to `redact`.

### Phase 6: Branding — mostly-no-pii → PIIvacy

- Realized the name limited mainstream appeal. Brainstormed: PIIvacy.
  PII embedded inside privacy. One word, sticky pun, reads naturally
  in sentences ("we use PIIvacy in front of every OpenAI call"), keeps
  the honest tagline ("Go ahead. Paste it in.") that already worked.
- Verified `piivacy` was free on npm (404 on the registry endpoint).
- Renamed everything in one pass:
  - Both GitHub repos: `mostly-no-pii` → `piivacy`,
    `mostly-no-pii-demo` → `piivacy-demo`.
  - Both local directories.
  - `package.json` names + the file: dependency.
  - Source-code error message prefixes (`piivacy: invalid mode...`).
  - Adapter referer/title strings.
  - All UI copy and the page title.
  - Vendor folder path.
  - Railway project + service rename via the GraphQL API
    (`projectUpdate`, `serviceUpdate` mutations).
  - Railway auto-domain rotated from `mostly-no-pii-demo-production...`
    to `piivacy-demo-production...` (delete old domain + create new).
- Local Railway CLI config moved to track the new directory path.
- Both GitHub repos pushed under their new names; old URLs auto-redirect
  via GitHub's repo-rename redirects.

### Phase 7: Custom domain piivacy.dev

- Domain registered and pointed at Cloudflare DNS.
- Custom domain attached to the Railway service via the API
  (returned the DNS records that needed adding).
- Cloudflare DNS records added (CNAME to the Railway edge hostname,
  set to "DNS only" not "Proxied" so Railway could issue Let's Encrypt).
- Verified.

### Phase 8: GitHub auto-deploy + cross-repo sync

- Old workflow: edit piivacy → `npm run vendor` in piivacy-demo → commit
  demo → push demo. Two repos to touch for one logical change.
- Switched piivacy-demo's package.json to use a git dependency:
  `"piivacy": "github:callieschneider/piivacy#main"`. Deleted the
  vendor/ directory. Each Railway build now `npm install`s a fresh
  clone of piivacy.
- Wired up Railway → GitHub for automatic redeploy on push to either
  repo's main branch.
- For the cross-repo case (push to piivacy → demo redeploys), added a
  GitHub Action workflow at `piivacy/.github/workflows/redeploy-demo.yml`.
  On every push to main, the workflow POSTs a `serviceInstanceRedeploy`
  GraphQL mutation to Railway, using a project-scoped token stored as a
  GitHub secret. End-to-end: ~6 seconds from `git push` to Railway build
  start, ~60-90 seconds total to live.
- Verified end-to-end with manual `gh workflow run`.

### Phase 9: SEO + AIO + contact form

- Full meta-tag pass on the index.html: refined description with keywords
  woven in, comprehensive keywords meta, canonical link, Open Graph
  block (og:title, description, url, image with width/height/alt,
  site_name, locale), Twitter Card with summary_large_image, theme-color,
  robots directive.
- JSON-LD structured data with three @types: SoftwareApplication, WebSite,
  and FAQPage with five Q&As that AI search engines (Perplexity, ChatGPT
  search, Claude search, Gemini) can index directly into answers.
- `/llms.txt` — AI-search-optimization markdown spec.
- `/robots.txt` — explicitly allows GPTBot, ChatGPT-User, Google-Extended,
  PerplexityBot, Claude-Web, anthropic-ai, cohere-ai, CCBot.
- `/sitemap.xml` — all anchor sections with priorities.
- **Contact form**: Resend-backed POST /api/contact with honeypot field
  + 5/hour-per-IP rate limit. Sends FROM `hi@piivacy.dev` (verified
  Resend domain via DKIM/SPF DNS records on Cloudflare) TO the user's
  hidden destination email (server-side env var). Form's `Reply-To` is
  the submitter's email so direct replies work without the recipient's
  address being exposed in HTML.

### Phase 10: Agent-discovery kit

- Three coordinated artifacts so AI agents (Cursor, Claude Code,
  Continue, Cline, Aider, Devin, Codex) can discover and integrate the
  package autonomously:
  - `/agents.md` — markdown agent guide with install + 7 recipes for
    common tasks + things-agents-get-wrong.
  - `/.well-known/agents.json` — structured manifest with API surface,
    modes, patterns, agent instructions block.
  - `/agents` — UI page with copy-paste recipes for every major coding
    agent (Cursor AGENTS.md, Claude CLAUDE.md, Continue config.json,
    Cline custom instructions, Aider --read flag, Codex CLI, generic).

### Phase 11: Heavy testing — 16k fuzz iterations

- Wrote `scripts/fuzz.mjs` — generative property-test runner with six
  scenarios:
  1. **roundtrip**: `restore(scrub(text, session))` === `text`
  2. **idempotence**: scrubbing twice with the same session produces the
     same output
  3. **multiturn**: shared values across turns get the same substitute,
     all turns round-trip
  4. **tokenLiteralPassthrough**: pre-existing or LLM-invented
     `[[LABEL_N]]` tokens aren't rewritten
  5. **manualSecret**: `registerSecret()` works for non-regex PII
  6. **inventoryDedup**: `listRedactions()` reports each value once with
     correct usage counts
- First fuzz run: 500 iterations → 32 failures (23 idempotence + 80
  multi-turn). Multi-turn was a test bug (`findSubstitute` returned the
  first match regardless of value). Idempotence was a real package bug.
- **Real bug 1 — idempotence**: a previous-pass realistic fake (e.g. ZIP
  fake `00001`, phone fake `(555) 010-0001`, email fake
  `redacted1@example.com`) itself matches the same regex and gets
  re-redacted on a second pass. Fix: scrub now skips values already in
  `session.fakes`.
- **Real bug 2 — IBAN regex over-greedy across separators**: the loose
  pattern `\b[A-Z]{2}\d{2}(?:[ -]?[A-Z0-9]){11,30}\b` allowed separators
  between any two characters and would extend a match into adjacent text
  ("GB82WEST...432 EB" eating a MAC's first byte). Validate then
  rejected the over-greedy match and the regex didn't backtrack to the
  correct shorter slice, so the IBAN was lost. Fix: stricter regex with
  consistent-separator group formats.
- **Real bug 3 — CC regex over-greedy across separators (same class)**:
  `\b(?:\d[ -]?){12,18}\d\b` would match across a ZIP+space+CC
  boundary, fail Luhn, and lose the real CC. Same kind of fix with
  `(?<![0-9])` + `(?![0-9])` lookarounds and consistent group formats.
- After fixes: 6,000 iterations across 6 seeds, plus a 10,000-iteration
  stress run with seed 0xCAFE — **zero failures** across all 16,000
  total fuzz iterations + 218 unit tests.

- `npm publish --access public`.

## Notable decisions and tradeoffs

| Decision | Why |
|---|---|
| Plain ESM, no TS | Stays dependency-free, ships smaller, no build step. Types are coming as a separate `.d.ts` later. |
| `node:test` not Vitest/Mocha | Zero devDependencies. Native to Node 18+. |
| Static name table not LLM at runtime | LLM at runtime would mean every realistic-mode scrub waits on a network call. The static table is deterministic and ships with the package. Adapter pattern is there for users who want LLM-based names. |
| Demographic bucketing + frequency, not embeddings | Embeddings cluster names by orthographic stem, producing spelling variants instead of demographically similar alternatives. Frequency rank within bucket gave the right answer (Marcus → Terrance not Marcus → Marquis). |
| Three modes not one | Token mode is restore-bulletproof but bad for LLM fluency. Realistic mode reads natural but restore needs longest-match. Pass-through is for when the LLM literally needs the value (addresses for local search). Different jobs. |
| Hard-coded safety override | Even if a misconfigured intent classifier says `secrets: preserve`, the package downgrades to `token`. No way to bypass. |
| Git dependency over npm dep for the demo | The demo can pick up any change to piivacy main without us bumping versions. Simpler workflow during heavy iteration. |
| GitHub Actions for cross-repo sync | One push to piivacy → demo redeploys automatically. No vendor copy to maintain. |
| Fuzz runner instead of just unit tests | 218 unit tests are exhaustive on intentional cases. Fuzz finds the unintentional ones — like idempotence failures from realistic fakes that themselves match regex. |

## Open questions / future work

- **Types**: ship `.d.ts` files for proper TypeScript editor experience.
- **Adapter library**: more name-substitution adapter options
  (HuggingFace inference endpoints, Vertex AI).
- **Non-Latin names**: extend the names table to Hangul, CJK, Arabic,
  Devanagari. Format is open; PRs welcome.
- **More patterns**: additional secret patterns as new providers emerge.
  Pattern registry is pluggable; users can add their own.
- **CLI**: optional `npx piivacy` binary that wraps a one-shot scrub /
  restore for command-line use.
- **Browser bundle**: the package works in browsers today via the
  WebLLM adapter, but a properly-bundled version with the names table
  pre-included as JSON would simplify drop-in browser use.
- **Self-host the diagrams**: gpt-image-2 generations are great, but
  hand-drawn SVGs would scale better at small sizes.
